API ENDPOINTS SUMMARY
================================================================================

Anomaly Detection:
  POST /api/anomaly/check-wallet
  Body: [{ wallet, amount, timestamp, tx_type, gas_price }]
  Response: { is_anomalous, anomaly_score, risk_level }

Fraud Detection:
  POST /api/fraud/check-loan
  Body: { wallet, amount, purpose, wallet_age_days, ... }
  Response: { is_fraudulent, fraud_score, risk_category, reasons }

Explainability:
  POST /api/explain/fraud-decision
  Body: { amount, wallet_age_days, ... }
  Response: { top_features, explanation_text, visualization }

RL Policy:
  POST /api/rl/recommend-action
  Body: { loan_request, current_state }
  Response: { action, confidence, expected_reward }

Attack Detection:
  POST /api/security/check-transaction
  Body: { tx_hash, call_sequence, gas_prices }
  Response: { is_attack, attack_type, probability }

Alerts:
  GET /api/alerts/recent?limit=10
  Response: [{ timestamp, type, severity, message }]

DEBUGGING TIPS
================================================================================

Issue: "Module not found"
→ Activate venv: source venv/bin/activate

Issue: "Model file not found"
→ Train models first: python scripts/train_*.py

Issue: "CUDA out of memory"
→ Use CPU: device='cpu' in PyTorch

Issue: "Frontend can't connect to ML backend"
→ Check CORS settings in FastAPI
→ Verify ML backend is running on port 8000

Issue: "Low model accuracy"
→ Generate more synthetic data
→ Tune hyperparameters
→ Check feature engineering

Issue: "RL agent not learning"
→ Increase training timesteps
→ Adjust reward function
→ Check environment is working

Q&A PREPARATION
================================================================================

Q: Why use ML instead of rule-based systems?
A: ML adapts to new patterns, rules are brittle and easy to bypass.

Q: How do you prevent adversarial attacks on ML models?
A: Ensemble methods, adversarial training, human-in-the-loop validation.

Q: What about privacy with ML models?
A: Only aggregated features used, no raw user data stored. Can extend to
   federated learning for distributed privacy.

Q: How scalable is this approach?
A: Current: 1000+ loans/day. Can scale horizontally with model serving
   (TorchServe) and caching (Redis).

Q: What's the biggest limitation?
A: Data quality. Synthetic data for demo. Real deployment needs historical
   transaction data for better training.

Q: How do you keep models updated?
A: Scheduled retraining (weekly/monthly), online learning, A/B testing.
