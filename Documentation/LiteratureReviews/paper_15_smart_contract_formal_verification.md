# Literature Review: Paper 15

**Paper Title:** A Survey of Smart Contract Formal Specification and Verification
**Paper Link:** https://dl.acm.org/doi/10.1145/3464421
**Authors:** Palina Tolmach, Yi Li, Shang-Wei Lin, Yang Liu
**Affiliation:** Nanyang Technological University, Singapore
**Source:** ACM Computing Surveys, 54(7), 2021

**Abstract:**
Smart contracts are programs stored on a blockchain that automatically execute when predetermined conditions are met. Since their invention, numerous methods for formal specification and verification of smart contracts have been proposed. This survey provides a comprehensive and systematic overview of these methods, covering specification languages, verification techniques, and tool implementations. The paper analyzes 42 tools and frameworks for smart contract verification, evaluating their coverage of vulnerability classes, scalability, false positive/negative rates, and practical usability for developers.

**Introduction:**
The immutability of deployed smart contracts fundamentally changes the cost of software bugs. In traditional software, patches can be deployed to fix vulnerabilities; in smart contracts, deployed code cannot be modified (unless proxy patterns are used, which introduce their own risks). This immutability means that every vulnerability discovered after deployment is permanently exploitable—the Parity wallet bug that froze $280 million in ETH is a permanent loss with no technical remedy. Formal verification, which provides mathematical guarantees of correctness with respect to a specification, offers the strongest form of assurance. However, formal methods have historically been confined to safety-critical systems (avionics, medical devices) due to their complexity and cost. The growth of DeFi—where contracts manage billions of dollars—has brought smart contracts into the category of software where formal verification is justified.

**Motivation:**
Informal testing (unit tests, integration tests, manual review) catches many bugs but cannot guarantee absence of vulnerabilities. Security audits by firms like OpenZeppelin, Trail of Bits, and Consensys Diligence provide expert review but are expensive ($50K–$500K per audit), time-limited, and depend on auditor expertise. Automated security tools (static analysis, symbolic execution) can find known vulnerability patterns but produce false positives and cannot detect novel attack vectors. Formal verification bridges this gap by mathematically proving that a contract satisfies its specification—but adoption requires understanding which techniques are mature, practical, and effective for DeFi contracts.

**Methodology:**
The survey systematically categorizes the field along three axes: **(1) Specification Languages** — how developers express what the contract should do: temporal logic (CTL, LTL for expressing properties like "a loan can never be approved twice"), Hoare logic (pre/postconditions for each function: "if balance ≥ amount, then after withdrawal, balance = old_balance − amount"), contract-level specifications (Solidity NatSpec, Scribble annotations), and domain-specific languages (Act, DeepSEA). **(2) Verification Techniques** — how correctness is checked: *model checking* (exhaustive state space exploration—effective for finite-state properties but suffers from state explosion), *symbolic execution* (executing programs with symbolic rather than concrete inputs—Mythril, Manticore—effective for path-based vulnerability detection), *abstract interpretation* (computing safe approximations of program behavior—Securify—fast but imprecise), *theorem proving* (interactive proof construction—KEVM, VeriSol—provides strongest guarantees but requires significant manual effort), *runtime verification* (monitoring deployed contracts for specification violations—useful for detecting attacks in real-time). **(3) Tool Implementations** — 42 tools are cataloged with their: target vulnerability classes, analysis approach, supported Solidity versions, false positive rates, performance benchmarks, and open-source availability. Key tools analyzed include: Mythril (symbolic execution, covers reentrancy/overflow/access control, open-source), Slither (static analysis, 40+ detectors, <1 second analysis time, high false positive rate), Securify (abstract interpretation, automated compliance checking), VeriSol (theorem proving for Solidity, requires Boogie intermediate language), and KEVM (formal semantics of EVM in the K framework, enables deductive verification).

**Results and Discussion:**
The survey reveals a clear trade-off between precision and automation: **symbolic execution tools** (Mythril, Manticore) are most effective for detecting reentrancy and arithmetic vulnerabilities, achieving 85–95% true positive rates, but require 30–300 seconds per contract and may miss vulnerabilities in large contracts due to path explosion. **Static analysis tools** (Slither) provide the fastest results (<1 second for most contracts) with broad vulnerability coverage (40+ detector patterns) but produce 15–30% false positive rates. **Theorem provers** (VeriSol, KEVM) offer complete mathematical guarantees but require 10–100× more developer effort than automated tools and are limited to small contract sizes. **The most effective strategy is multi-tool verification**: using Slither for initial fast screening, Mythril/Manticore for deeper path analysis of flagged areas, and formal provers for critical financial logic (e.g., the core loan approval and fund transfer functions). Runtime verification provides an additional layer for deployed contracts—monitoring execution traces for specification violations and triggering alerts or circuit breakers.

**Conclusion:**
The survey establishes that no single tool provides complete smart contract security—a multi-tool approach combining fast static analysis, thorough symbolic execution, and targeted formal proving provides the strongest practical assurance. For the Crypto World Bank, this informs the planned security verification pipeline: (1) Slither for continuous static analysis during development, (2) Mythril for symbolic execution before deployment, (3) targeted formal verification for critical functions (depositToReserve, approveLoan, transfer logic), and (4) runtime monitoring for deployed contracts. The survey's finding that arithmetic overflow vulnerabilities are eliminated by Solidity 0.8.x validates the project's choice of Solidity 0.8.20.
