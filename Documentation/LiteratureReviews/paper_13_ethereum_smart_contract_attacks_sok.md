# Literature Review: Paper 13

**Paper Title:** A Survey of Attacks on Ethereum Smart Contracts (SoK)
**Paper Link:** https://link.springer.com/chapter/10.1007/978-3-662-54455-6_8
**Authors:** Nicola Atzei, Massimo Bartoletti, Tiziana Cimoli
**Affiliation:** University of Cagliari, Italy
**Source:** Principles of Security and Trust (POST), Springer, 2017

**Abstract:**
Smart contracts are computer programs that can be correctly executed by a network of mutually distrusting nodes, without the need of an external trusted authority. Since smart contracts handle and transfer assets of considerable value, besides the blockchain infrastructure itself, they are attractive targets for attacks. This paper provides a comprehensive survey of Ethereum smart contract vulnerabilities, organized by the level at which they manifest (Solidity, EVM bytecode, blockchain), with analysis of real-world attack instances including the notorious DAO attack that resulted in a loss of 3.6 million ETH.

**Introduction:**
Ethereum, launched in 2015, extends blockchain technology beyond simple value transfers to general-purpose computation through smart contracts—self-executing programs deployed on the blockchain. By 2017, thousands of smart contracts managed billions of dollars in value. The DAO (Decentralized Autonomous Organization), a smart contract-based investment fund, accumulated over $150 million in ETH before a reentrancy vulnerability was exploited in June 2016, draining 3.6 million ETH (~$50 million at the time). This attack was so severe it led to a hard fork of the Ethereum blockchain (creating Ethereum and Ethereum Classic). Despite this high-profile failure, the smart contract ecosystem continued growing without comprehensive security systematization—this paper fills that gap.

**Motivation:**
Smart contracts differ from traditional software in critical ways that amplify vulnerability impact: (1) **immutability** — once deployed, code cannot be patched; bugs persist forever unless proxy patterns are used; (2) **financial exposure** — contracts directly control assets, so any vulnerability is immediately exploitable for financial gain; (3) **adversarial environment** — anyone can interact with a deployed contract, including sophisticated attackers; (4) **semantic gaps** — developers write in high-level Solidity, but the EVM executes low-level bytecode, and the blockchain environment adds additional execution semantics (gas, block timestamps, transaction ordering) that create vulnerability classes not present in traditional software. A systematic taxonomy is needed to enable proactive security design.

**Methodology:**
The survey analyzes Ethereum smart contracts at **three abstraction levels**: **(1) Solidity Level** — vulnerabilities arising from Solidity language semantics and developer misunderstanding: *reentrancy* (external calls to untrusted contracts before state updates, enabling recursive callbacks—the DAO exploit); *gasless send* (transfer() and send() have 2,300 gas stipend, causing silent failure for contracts with fallback functions); *exception disorder* (inconsistent exception propagation—some calls revert state while others return false); *type casting* (unintended truncation when converting between types); *access control* (public functions without proper modifiers enabling unauthorized access). **(2) EVM Level** — vulnerabilities from EVM bytecode execution: *integer overflow/underflow* (uint256 wrapping—adding 1 to max value gives 0, subtracting 1 from 0 gives max value; addressed in Solidity 0.8.x with built-in checks); *stack size limit* (call stack depth of 1024 enabling stack depth attacks); *immutable bugs* (deployed bytecode cannot be modified). **(3) Blockchain Level** — vulnerabilities from the blockchain execution environment: *timestamp dependence* (miners can manipulate block.timestamp within ~900 seconds, affecting time-dependent logic); *transaction ordering dependence* (miners can reorder transactions within a block for profit—front-running); *unpredictable state* (contract state can change between transaction submission and execution). For each vulnerability, the paper provides: formal definition, attack mechanism, real-world instances, and mitigation strategies.

**Results and Discussion:**
**Reentrancy** remains the most financially damaging vulnerability class—the DAO attack exploited the pattern of `msg.sender.call.value(amount)()` before updating the sender's balance, enabling recursive withdrawal. The mitigation—checks-effects-interactions pattern—is now standard practice, codified in OpenZeppelin's ReentrancyGuard modifier. **Integer overflow/underflow** was historically critical (e.g., BEC token attack where overflow created billions of tokens from nothing), but Solidity 0.8.x eliminates this entire class with built-in overflow checks. **Access control violations** through missing or incorrect modifiers enable unauthorized function calls—for example, the Parity wallet freeze ($280 million) occurred when a developer accidentally called an unprotected initialization function. **Transaction ordering dependence** (front-running) is systemic on Ethereum—MEV (Miner Extractable Value) now exceeds $600M+ annually, exploiting this vulnerability class. The survey finds that most vulnerabilities stem from the **semantic gap** between developer intention (high-level Solidity logic) and actual execution (EVM bytecode + blockchain environment semantics).

**Conclusion:**
This vulnerability taxonomy is the foundational security reference for Ethereum smart contract development. The Crypto World Bank's smart contract suite (WorldBankReserve.sol, NationalBank.sol, LocalBank.sol) directly implements mitigations for every vulnerability class identified: OpenZeppelin's **ReentrancyGuard** prevents reentrancy, Solidity **0.8.20's built-in overflow protection** eliminates integer overflow/underflow, **Ownable** and custom hierarchical modifiers enforce access control, and explicit **require()** validation at every state-changing function entry point prevents unauthorized execution. The checks-effects-interactions pattern is followed throughout.
